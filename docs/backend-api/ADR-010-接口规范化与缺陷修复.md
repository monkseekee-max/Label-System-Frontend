开发人员在实际联调过程中发现了多个接口层面的问题，涵盖参数设计、响应格式、错误处理、幂等性、认证一致性等方面。本 ADR 针对每个问题给出决策和修复方案，确保所有接口符合 api.md 文档规范并具备生产可用性。

## BUG-001: `create_benchmark` 使用 Query 参数而非 JSON Body

### 上下文

当前 `POST /api/v1/eval/benchmark` 使用 Query 参数接收 `name`、`description`、`samples`。但 `samples` 是一个复杂的列表结构，不适合通过 URL query 传递。且 api.md 文档 §6.3 明确要求使用 JSON Body。

**当前代码**:
```python
@app.post("/api/v1/eval/benchmark")
async def create_benchmark(name: str, description: str = "",
                            samples: List[Dict] = None, session=Depends(get_session)):
```

### 决策

创建 `BenchmarkCreateRequest` Pydantic 模型 (已在模块顶部定义但未使用)，改用 JSON Body 接收请求。

**修复后**:
```python
@app.post("/api/v1/eval/benchmark")
async def create_benchmark(req: BenchmarkCreateRequest, session=Depends(get_session)):
```

### 理由

1. API 设计最佳实践：POST 创建资源应使用 JSON Body
2. `samples` 是复杂嵌套结构，无法通过 URL query 传递
3. api.md 文档已明确要求 JSON Body
4. `BenchmarkCreateRequest` 模型已定义但未使用

## BUG-002: `data_sync` 接口参数语义不清

### 上下文

当前 `POST /api/v1/data/sync` 接口同时接收 `source` 作为 Query 参数和可选的 `DataImportRequest` Body。当 `source=labelstudio` 时需要 `project_id` 参数，但 `project_id` 也在 Query 中。整个接口设计混乱。

### 决策

保持 `source` 为 Query 参数（便于快速区分模式），但增加 `project_id` 到 Query 参数明确化，并改善文档说明。

**修复后**:
```python
@app.post("/api/v1/data/sync")
async def data_sync(
    source: str = Query("labelstudio", pattern="^(labelstudio|upload)$"),
    project_id: int = Query(1, description="Label Studio 项目 ID (source=labelstudio时)"),
    req: Optional[DataImportRequest] = None,
    session=Depends(get_session),
):
```

同时增加参数校验：当 `source=upload` 时必须提供 `req`。

### 理由

1. `source` 作为路由选择器放在 Query 是合理的
2. `project_id` 应该有明确的文档说明
3. 参数组合校验避免无效请求

## BUG-003: 响应时间戳格式不一致

### 上下文

部分端点返回的时间戳包含时区信息（如 `2026-06-02T14:15:30+08:00`），部分不包含（如 `2026-06-02T14:15:30`）。api.md §1.3 明确要求 ISO 8601 + 时区。

**不一致示例**:
- `data_status.last_import_at` — 取决于 PG 返回值
- `training_metrics.started_at` — 使用 `isoformat()` 但 PG 无时区数据时不带时区
- `system_info` 中无时间字段

### 决策

1. 所有 DateTime 字段使用 `DateTime(timezone=True)` — 已在 models.py 中实施
2. 创建统一的时间格式化工具函数 `format_dt()`，确保所有端点返回一致的 ISO 8601 + 时区
3. 对 SQLite 测试环境，手动附加 UTC 时区

**新增工具函数**:
```python
def format_dt(dt) -> Optional[str]:
    """统一格式化时间为 ISO 8601 + UTC"""
    if dt is None:
        return None
    if dt.tzinfo is None:
        dt = dt.replace(tzinfo=timezone.utc)
    return dt.isoformat()
```

### 理由

1. api.md §1.3 明确要求 ISO 8601 + 时区
2. 前端解析时区无关的时间戳可能出错
3. 统一格式减少前端适配负担

## BUG-004: 分页参数默认值不统一

### 上下文

不同端点的分页参数不一致：
- `data_list`: `limit=50, le=500`
- `list_eval_results`: `limit=50, le=200`
- `media_list`: `limit=50, le=500`
- `list_tasks`: `limit=50, le=200`

api.md §1.5 约定: `limit` 默认 20, 范围 1-100。

### 决策

统一所有分页接口：`limit` 默认 20, 上限 100；`offset` 默认 0。

### 理由

1. api.md 是接口契约，代码必须与文档一致
2. 默认 20 条减少首次加载时间
3. 上限 100 防止过大查询压垮服务

## BUG-005: `data_import` 未正确从 records 提取字段

### 上下文

`DataImportRequest.records` 是 `List[Dict[str, Any]]`，但 `IngestPipeline.ingest_records()` 可能未正确处理输入 record 中各种字段格式。实际标注数据有 `qa_content.records[]` 嵌套结构（见 api.md §4.1），但 IngestPipeline 期望的是平铺的 `{question, answer, source}` 格式。

### 决策

1. 在 `IngestPipeline` 中增加格式检测和自动转换
2. 支持 api.md §4.1 中定义的两种格式：
   - 平铺格式: `{question, answer, source_segment, ...}`
   - 嵌套格式: `{task_id, resource_name, qa_content: {records: [...]}}`
3. 增加字段映射和默认值填充

### 理由

1. 外部 ETL 管道的数据格式不可控，需要适配
2. 嵌套格式是 api.md 文档中的标准格式
3. 缺少 annotation_id 会导致数据无法溯源

## BUG-006: 错误响应格式不统一

### 上下文

api.md §11.2 定义了标准错误格式：
```json
{
  "detail": "...",
  "error_code": "...",
  "context": {...}
}
```

但当前大部分端点仅返回 `{"detail": "..."}` 或直接抛出 HTTPException，缺少 `error_code` 和 `context`。

### 决策

1. 创建 `ErrorResponse` 模型
2. 创建自定义异常类 `AppException`，携带 `error_code` 和 `context`
3. 增加全局异常处理器，统一格式化错误响应
4. 关键端点（data_import, eval_run, train_text）使用结构化错误

**不改动**：FastAPI 内置的 422 Validation Error 格式（保持原生行为）

### 理由

1. 前端需要 `error_code` 做国际化错误提示
2. `context` 提供调试信息，减少沟通成本
3. 统一格式是 API 设计基本原则

## BUG-007: `scheduler/trigger` 与 `train/text` 功能重复

### 上下文

`POST /api/v1/scheduler/trigger` 和 `POST /api/v1/train/text` 都触发文本 LoRA 训练，参数几乎完全相同（model_key, lora_profile, training_profile, use_seed, dataset_path）。两者都调用 `get_dispatcher().trigger_text_training()`。

### 决策

1. `scheduler/trigger` 作为「带调度语义」的触发入口（可能受调度模式限制）
2. `train/text` 作为「直接触发」入口（不受调度模式限制）
3. 两者内部统一调用 `training_dispatcher`，但 `scheduler/trigger` 增加调度模式检查
4. 文档中明确两者的区别和使用场景

### 理由

1. 两个端点有不同语义：一个是调度触发，一个是直接触发
2. 保留两个入口便于不同使用场景
3. 但需要明确行为差异

## BUG-008: `inference/chat` 无超时配置和重试机制

### 上下文

当前 `inference/chat` 直接调用 `vllm_inference()`，硬编码 120 秒超时。如果 vLLM 正在重启或暂时不可用，请求直接失败返回 503，无重试机制。

### 决策

1. 增加可配置的超时参数（默认 60s）
2. 增加 1 次自动重试（间隔 2s）
3. 增加 vLLM 就绪检查（调用前先检查 `/v1/models`）
4. 超时和重试配置通过 `InferenceRequest` 可选参数控制

### 理由

1. vLLM 重启期间（约 30s）推理不可用，自动重试可覆盖大部分场景
2. 硬编码 120s 过长，用户应可控制
3. 就绪检查避免无谓等待

## BUG-009: `data_update` 缺少字段校验和 `updated_at` 返回

### 上下文

`PUT /api/v1/data/{record_id}` 直接接受 `Dict[str, Any]`，无字段白名单校验。同时响应中缺少 `updated_at` 字段。

### 决策

1. 创建 `DataUpdateRequest` Pydantic 模型，仅允许白名单字段
2. 响应中包含 `updated_at`
3. 状态变更时校验合法值 (collected/cleaned/skipped)

### 理由

1. 无白名单校验可能导致意外字段被写入
2. `updated_at` 是前端刷新列表的关键字段
3. 状态值应受限于合法枚举

## BUG-010: WebSocket 鉴权使用 query param 明文传递 token

### 上下文

浏览器 WebSocket API 不支持自定义 Header，因此当前通过 `?token=xxx` 在 URL 中传递 API Key。这在浏览器历史和服务器日志中会暴露密钥。

### 决策

1. MVP 阶段保持 `?token=` 方案（已实现且可用）
2. 增加 ticket 机制：先通过 REST API `/api/v1/ws/ticket` 获取一次性短时 ticket（30s 有效），然后用 `?ticket=xxx` 连接 WebSocket
3. ticket 使用后立即失效

**优先级**: P2（当前方案功能可用，安全增强为后续优化）

### 理由

1. 完全消除 WebSocket 鉴权问题需要较大改动
2. ticket 机制是业界标准做法
3. MVP 阶段功能优先，安全加固可后续迭代

## BUG-011: `eval/benchmark` POST 不持久化 samples

### 上下文

`create_benchmark` 端点接收 `samples` 列表但仅将 `sample_count` 存入数据库，不实际持久化 sample 数据。后续 `benchmark_detail` 读取文件时会找不到数据。

### 决策

将 samples 数据写入 `{file_path}` 指定的 JSONL 文件，确保 benchmark_detail 能正确读取。

### 理由

1. 不持久化 samples 导致 benchmark 详情查询为空
2. JSONL 格式与项目数据管道一致
3. 文件持久化后 benchmark 可复用

## BUG-012: `lora_activate` 缺少原子性保证

### 上下文

`lora_activate` 先更新 DB（is_active 标记），再更新 symlink，再重启 vLLM。如果 symlink 更新失败或 vLLM 重启失败，DB 和实际状态不一致。

### 决策

1. 在 `activate` 方法中增加 try-except 回滚逻辑
2. symlink 更新失败时回滚 DB 状态
3. vLLM 重启失败不影响 activate（记录警告），因为推理可用性不影响版本管理
4. 增加 `/api/v1/lora/active` 与 DB 状态的一致性校验端点

### 理由

1. DB 和文件系统不一致会导致版本管理混乱
2. 回滚逻辑是分布式操作的基本要求
3. vLLM 重启是独立关注点，不应阻塞版本激活

## BUG-013: POST 创建资源缺少 201 状态码

### 上下文

多个 POST 创建端点返回 200 OK 而非 201 Created：
- `POST /api/v1/data/import`
- `POST /api/v1/eval/benchmark`
- `POST /api/v1/alignment/feedback`
- `POST /api/v1/media/upload`

### 决策

对创建新资源的 POST 端点返回 `201 Created`，使用 `Response(status_code=201)` 或在路由装饰器中指定 `status_code=201`。

### 理由

1. REST 语义：创建资源应返回 201
2. 前端可根据状态码区分「创建成功」和「查询成功」
3. api.md §11.1 中列出了 201 状态码但未在实现中使用

## BUG-014: `data_list` 缺少 sort 参数

### 上下文

`GET /api/v1/data/list` 固定按 `created_at DESC` 排序，不支持自定义排序。前端数据审核页面需要按质量评分、状态等字段排序。

### 决策

增加 `sort` 查询参数，支持 `created_at`、`quality_score`、`status` 字段排序，默认 `created_at DESC`。

### 理由

1. 数据审核需要多种排序视角
2. 质量分排序便于优先处理低质量数据
3. 实现简单，仅需动态构建 SQLAlchemy order_by

## BUG-015: 健康检查缺少依赖服务状态

### 上下文

`GET /health` 仅返回 `{"status": "ok"}`，不检查数据库连接、vLLM 状态等依赖服务。在数据库或 vLLM 不可用时仍返回 ok，对监控系统无价值。

### 决策

1. 增加数据库连接检查（简单 SELECT 1）
2. 可选检查 vLLM 状态（不阻塞健康检查）
3. 增加 `mode` 参数：`light`（默认，仅 DB）和 `full`（含 vLLM）

### 理由

1. 健康检查应反映服务真实状态
2. DB 不可用时 API 实质不可用
3. mode 参数避免健康检查过于重量级
