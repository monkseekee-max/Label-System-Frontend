# ADR-020: 前端↔后端集成契约对齐与鉴权统一

> **项目**: 智能标注系统 — 本地模型训练工厂
> **状态**: 已批准（P0-1/P0-3 已修复并 TDD 验证；D2/D4 列入后续演进）
> **日期**: 2026-06-18
> **决策者**: 项目组
> **触发**: `/platform/companies` 管理页 `{"detail":"无效的API Key"}` 缺陷 → 全量诊断前后端集成

---

## 0. 背景与动机

用户反馈「新账户注册必须选所属公司，但无从管理公司」。排查发现前端管理页
`/platform/companies` 代码完整（CRUD 齐全、路由已注册），却拉不到数据。由此对
**前端（react-antd-admin）↔ 后端（llm-factory FastAPI + label-system proxy）** 的集成契约
做了一次全量诊断，发现这是一类系统性问题，不止公司管理一处。

本文档记录诊断结论、已落地的 P0 修复、根因分析，以及后续契约对齐的决策方向。

---

## 1. 系统现状（集成拓扑）

```
浏览器(react-antd-admin, :3335/app.fun-md.com)
   │  ky request, prefixUrl=/api
   │  拦截器: Authorization: Bearer <token>  +  X-Lang  (+ 本次新增 X-API-Key)
   ▼
Vite dev proxy  /api → http://127.0.0.1:9090   (ADR-011)
   ▼
llm-factory FastAPI (:9090)
   ├── src/app/**           域 API (/api/v1/data|billing|eval|lora|label|quality|...)
   │                        鉴权: Depends(verify_api_key)
   ├── src/integration/label_system_proxy.py   代理 → label-system 微服务
   │                        (/api/auth /api/admin /api/platform /api/data /api/ai
   │                         /api/intelligence /api/training /api/workflows /api/v1/kb)
   │                        鉴权: extract_bearer_or_api_key (admin/* 前缀强制)
   └── src/common/security.py::verify_api_key  统一入口 verify_any_token
label-system (services/iam)  签发 JWT (username+password, 自有 secret)
```

**两套后端、两套鉴权、两套响应包壳** —— 这是所有集成裂缝的根源。

## 2. 诊断结论（按严重度）

### P0-1 ｜ platform 管理页鉴权断裂【已修复】

| 项 | 内容 |
|---|---|
| 现象 | `/platform/companies` 页 `{"detail":"无效的API Key"}`，列表空、建不了 |
| 根因 | 前端请求拦截器（`frontend/src/utils/request/index.ts`）只发 `Authorization: Bearer`，**不发 `X-API-Key`**；后端 `verify_api_key` 在 Bearer 校验失败时回退依赖 X-API-Key |
| 影响面 | 所有受 `verify_api_key` 保护且 Bearer 不被识别的端点：`platform/*`、`admin/*`、`v1/billing/*`、`v1/data/*`、`v1/eval/*`、`v1/lora/*` 等 |
| 修复 | `constants.ts` 新增 `API_KEY_HEADER`；`request/index.ts` 拦截器按 `VITE_LLM_FACTORY_API_KEY` 注入 `X-API-Key`；`.env` 配入 key |
| 验证 | Vite 已注入 env（`"VITE_LLM_FACTORY_API_KEY":"<redacted>"`）；后端实测 `GET/POST /api/platform/companies` 返回 200 |
| 安全注 | X-API-Key 经 VITE_ 变量打包进前端 bundle，**生产不可用裸 dev key**（见 P0-2 正解） |

### P0-2 ｜ ~~JWT 签发/校验密钥分治~~ → **实际为路由桩未接服务（已修正诊断）**

`verify_api_key`（`src/common/security.py`）实为统一入口，委托 `verify_any_token`
（`src/app/auth/token.py:127`），支持三轨：`Bearer <JWT>` / `Bearer <api_key>` /
`X-API-Key`，由 `allow_legacy_api_key = True`（`config.py:132`）放行。

**初判（已证伪）**：曾怀疑 iam 与 llm-factory 的 JWT secret 不一致导致 Bearer 失败。
**复查（确认）**：生产登录路由 `/api/auth/login`（`api_server.py:431`）签发与校验**同源**
（均用 `issue_human_tokens` + `settings.jwt_secret`），密钥**不存在**分治。真正病根见 P0-3。

### P0-3 ｜ 登录 / 公司路由为桩，未接真实服务【根因，已修复】

| 项 | 内容 |
|---|---|
| 现象 | 无法用真实账号登录；`auth/companies` 恒返回硬编码 HMJC；注册的用户登不进 |
| 根因 | `/api/auth/login`（`api_server.py:431`）是**桩**：`password != settings.api_key` 即拒，**未调用** `RegistrationService.login()`；`/api/auth/companies`（`api_server.py:422`）恒返回硬编码 `_AUTH_COMPANY`。而 `RegistrationService`（`src/app/auth/registration.py`，PG `label_users` 真实校验、PBKDF2 哈希）已实现且经 `test_registration.py` 验证——仅生产路由没接 |
| 修复 | login 路由改为：优先 `RegistrationService.login(phone, companyCode, password)`；**仅当「用户不存在」**才回退 `password==api_key` 引导通道（真实用户凭证错绝不回退，防 api_key 绕过真实账号）；companies 路由改为查真实 `label_companies(ENABLED)`，空表回退默认 |
| 验证 | TDD：`tests/unit/test_auth_real_login_wiring.py` 6 例（真实登录/错密码/api_key 兼容/JWT 访问受保护接口/真实公司/空表回退）全绿；连带 `test_auth_login_jwt` `test_login_tenant_injection` `test_registration` 共 25 例全绿 |
| 影响 | D3（统一 JWT secret）**取消**——同源无需统一；D1（Bearer 闭环）**达成**——真实登录签发的 JWT 可访问受保护接口 |

### P1-1 ｜ 响应包壳不统一

| 来源 | 形状 |
|---|---|
| llm-factory 业务 | `{code, message, data}` |
| FastAPI 错误 | `{detail}` |
| 部分域 | 裸 dict |
| 前端 `api/platform/companies.ts` | `toApiResponse` 把 `data → result` |
| 前端 `api/user` 鉴权流 | `AuthApiResponse` 期望 `result` |

后端 `GET /api/auth/companies` 实测返回 `{code,message,data:[...]}`，但前端鉴权层
按 `result` 取值 → 解析得 `undefined` → 登录页公司下拉曾表现为空。多处 `toApiResponse`
各自实现，口径不一。

### P1-2 ｜ 多后端路由前缀与双轨鉴权

前端业务端点经 **label_system_proxy**（`/api/auth` `/api/admin` `/api/platform`
`/api/data` `/api/ai` `/api/intelligence` `/api/training` `/api/workflows`
`/api/v1/kb`），llm-factory 域 API 走 `/api/v1/*`。两套前缀、两套鉴权
（proxy `extract_bearer_or_api_key` vs 域 `verify_api_key`）、两套包壳，认知与维护成本高。

### P1-3 ｜ 登录流程字段错位

| 层 | 字段 |
|---|---|
| 前端表单（`password-login.tsx`） | `phone` + `companyCode` + `password` |
| iam 后端（`services/iam/app/routes.py:193`） | `username` + `password` |
| `iam-user-mapping.ts` | 硬编码 `DEFAULT_COMPANY={HMJC,"华明检测"}`，`phone = email` |
| `GET /api/auth/companies` | 对**任意** phone 返回 HMJC |

多租户登录实为单租户：所有用户被映射到硬编码的「华明检测」。新公司需通过
`POST /api/platform/companies`（带 X-API-Key）创建，但 `auth/companies` 目前不按
phone 过滤，注册下拉恒为 HMJC。

### P2-1 ｜ mock/fake 覆盖缺口

`frontend/fake/*.ts` 仅覆盖模板 demo 路由：`/login` `/logout` `/refresh-token`
`/user-info` `/get-async-routes` `/role-list` `/menu-list` `/home*` `/notifications`
`/country-calling-codes` `/upload`。

真实业务端点（`auth/companies`、`platform/companies`、`admin/*`、`v1/*`、
`intelligence/*`、`training/*`）**均未 fake**，dev 期依赖真实 :9090 后端或
`VITE_ENABLE_MOCK_FALLBACK` 兜底（生产强制关闭）。后端未起即白屏/空数据。

## 4. 决策（契约对齐方向）

| 编号 | 决策 | 优先级 |
|---|---|---|
| **D1** | **鉴权统一为登录态 JWT**：受保护 UI 路由用 `require_role(min_role)`（基于 `Principal`），`X-API-Key` 仅保留给服务间/机器调用，不进前端 bundle | P1 |
| **D2** | **统一响应信封**：全后端 `{code, message, data}`；错误统一 `{code, message, data:null}`（不裸用 FastAPI `{detail}`）；前端单一 `toApiResponse`（`data→result`） | P1 |
| **D3** | ~~统一 JWT secret~~ —— **取消**：复查证实签发/校验同源（`issue_human_tokens`+`settings.jwt_secret`），不存在密钥分治；原 P0-2 诊断已证伪 | 取消 |
| **D4** | **收敛路由命名空间**：单一 `/api` 前缀，proxy 透明转发，前端无需区分 `v1/*` vs `platform/*` 的鉴权差异（大范围跨切面重构，列后续演进） | P2 后续 |
| **D5** | **登录/公司路由接通真实服务**（已落地）：`/api/auth/login` 优先 `RegistrationService.login`，仅「用户不存在」回退 api_key；`/api/auth/companies` 查真实 `label_companies` + 空表回退。注册→真实密码登录→Bearer 访问受保护接口全链路打通 | ✅ 已完成 |
| **D6** | **关键业务端点补 e2e 契约**：对 `auth/login` `auth/companies` `platform/companies` 等加前后端契约测试；dev 补必要 fake，避免后端未起即白屏 | P2 |

## 5. 风险与回滚

- **R1（安全）**：当前修复把 dev `X-API-Key` 打包进前端 bundle，生产若误用则泄露。
  缓解：生产构建强制 `VITE_LLM_FACTORY_API_KEY` 留空并依赖 D1/D3 的 JWT；config.py
  生产校验器（`config.py:171`）已禁止默认 key/secret，沿用同一防线。
- **R2（兼容）**：D1 改 `require_role` 后，既有裸 `X-API-Key` 服务间调用需保留兼容通道
  （`allow_legacy_api_key` 过渡期保持 True，最终收敛为机器 Principal）。
- **回滚**：P0-1 修复为纯前端增量（env + 拦截器），移除 `VITE_LLM_FACTORY_API_KEY` 即
  回到原状，无后端/数据副作用。
